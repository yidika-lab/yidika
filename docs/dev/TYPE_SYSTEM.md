 PhilosophieLe système de types de Yidika est statique, explicite (par défaut), et typé par valeur. Il vise à éliminer les erreurs de dépassement (buffer overflow) et les comportements indéfinis, tout en permettant un contrôle fin de la mémoire (int8 vs int).2.2 Taxonomie des TypesCatégorieTypesDescriptionEntiersint, rint, int<N>int (natif), rint (signé), int8 à int64Réelsreal, real<N>real (natif), real16 (half), real32, real64ComplexescomplexNombres complexes $(a + bi)$Logiquesbooltrue, falseTextuelsstrChaîne de caractères immuableSymbolessymbolType énuméré léger (@, "f")Non-valeursNone, nullNone (absence d'objet), null (zéro mémoire)2.3 Types Composés et StructuresVector (vector) : Structure de données contiguë.x:y (2D), x:y:z (3D).Optimisation : Le compilateur transforme ces structures en vecteurs SIMD (AVX/NEON) pour le CPU ou le GPU.Matrix (matrix) : Grille de données ([x:y], [x:y]).Utilisation privilégiée : Calculs haute performance et NPU.Listes & Sets ([], {}) :name:[<type>] : Liste dynamique.name:{<type>} : Ensemble unique.Struct & Union :struct : Alignement mémoire strict (pas de padding implicite sauf directive).union : Type à taille variable (taille de l'élément le plus grand).2.4 Inférence et Union TypesYidika supporte les Union Types pour plus de flexibilité sans sacrifier la sécurité :Extrait de codetype User = int | str; // Le compilateur génère une balise (tag) pour identifier le type à l'exécution.
Règle de sécurité : Toute opération sur une union nécessite un bloc de contrôle (if/match) pour vérifier le type actif, garantissant ainsi l'absence d'accès mémoire invalide.2.5 Règles de Typage (Type Safety)Explicitation : name:<type> = <value>. Si le type est omis, le compilateur tente l'inférence. Si l'inférence est ambiguë, une erreur de compilation est levée.Conversion :Conversion implicite : Tolérée uniquement si elle ne génère aucune perte de précision (ex: int8 vers int64).Conversion explicite : Requise via l'opérateur as pour tout autre cas.Null-Safety : null est une valeur binaire (0). None est une instruction logique. Le compilateur interdit l'accès à un membre d'un objet pouvant être None sans vérification préalable.2.6 Interopérabilité (FFI Types)Pour le support c, c++, rust, le typage respecte l'ABI (Application Binary Interface) de la plateforme :Les types <type><byte> (ex: int32) sont mappés directement sur les types C standard (int32_t).Le typage Yidika "force" l'alignement mémoire pour correspondre exactement à ce que la bibliothèque C/C++ attend.2.7 Exemple de validation de type (AST-level)Pour l'expression x:int = 10, le compilateur effectue :Look-up : Vérifie si int est défini dans le scope.Constraint : Vérifie que 10 respecte les bornes de int (définies dans ARCHITECTURE.md).Bordure : Si 10 est assigné à int8, vérifie 10 <= 127.C'est une spécification rigoureuse qui protège votre runtime. Avec ce TYPE_SYSTEM.md, vous définissez la logique qui empêchera les développeurs de faire des erreurs de segmentation.
